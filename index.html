Guess the Verb

A Parser Puzzle Laboratory — built with Sharpee

Play Source

Explore your grandmother's old house and find the family heirloom she left behind. Six rooms, ten puzzles, and every room tests a different verb pattern.

This game is also a laboratory for reducing guess-the-verb frustration — the #1 pain point in parser-based interactive fiction. It uses every layer of Sharpee's command pipeline to make natural language input just work.

The Problem

In traditional parser IF, you know what you want to do but the game won't accept how you say it. The doormat description says "something glints underneath" — you type check under mat, lift mat, search mat. The game says "I don't understand that." The correct answer was look under mat. You didn't fail a puzzle. You failed to guess the verb.

This project tackles that problem head-on by intervening at every stage of Sharpee's command pipeline.

The Layers

Sharpee processes a command through a pipeline: Parse → Validate → Execute → Report. Each stage has story-level extension points. Disambiguation fixes go where they naturally belong.

Pipeline StageExtension PointWhat It Fixes
Parse: VocabularyregisterVocabulary()Word-level synonyms before grammar matching
Parse: GrammargetStoryGrammar().define()Structural patterns, question forms, multi-word verbs
Validate: EntitiesIdentityTrait aliasesNoun resolution — more ways to refer to things
Execute: ActionsgetCustomActions()New verbs the stdlib doesn't have
Execute: InterceptorsregisterActionInterceptor()Redirect standard actions with custom responses
Post-turn: PluginsTurnPlugin.onAfterAction()State tracking (e.g., remembering last room for GO BACK)
Display: ErrorshandleStoryEventParse-error-specific guidance instead of generic "I don't understand"

Examples

Vocabulary — check and read become examine synonyms. The parser recognizes them as verbs before grammar matching even starts. Type read recipe card and the kitchen recipe becomes examinable.

Grammar — where am i is a question, not a verb-noun command. A literal grammar rule maps it directly to a custom action. what do i have routes straight to the stdlib inventory action. flip through books maps to examine. go upstairs bypasses the direction system entirely with a literal rule.

Entity aliases — The overcoat responds to coat, jacket, wool coat, pockets. The doorbell responds to bell, button, buzzer. More aliases means fewer "I don't see that here" errors.

Custom actions — go back returns to the previous room. The stdlib doesn't track room history, so a custom action reads last-room from world state and moves the player. light fireplace routes to a custom burning action because the stdlib doesn't have one.

Interceptors — take mat doesn't fail with "that's scenery." Instead, the taking interceptor blocks the take but reveals the hidden key underneath — the action the player actually wanted.

Plugins — A movement-tracking plugin runs after every successful GO action, saving the origin room to world state. This powers the go back action without polluting action code.

Error messages — When nothing matches, the game distinguishes between "I don't know that word" (unknown verb), "You don't see that here" (entity not in scope), and "I don't understand that phrasing" (syntax error). Each points the player toward VERBS, LOOK, or HELP.

Naive Walkthrough

The entire game is completable using natural language — no parser-speak, no abbreviations, no two-word commands required. This walkthrough uses only phrases a first-time player might actually type:

where am I → look around → examine the doormat → look under the mat → pick up the key → unlock the door with the key → open the door → go inside → examine the overcoat → search the pockets → head east → open the desk → take the small key → move the painting → light the fireplace → examine the hearthstone → open the safe → take the spring → leave → go south → read the recipe card → head south → grab the trowel → dig the flower bed → grab the gears → go back inside → go north → go upstairs → put the gears in the music box → put the spring in the music box → wind the music box → open the trunk → take the locket

The Door Pattern

The locked front door demonstrates Sharpee's standard door/key system. Five traits work together: IdentityTrait (name and aliases), DoorTrait (connects two rooms), OpenableTrait (open/close), LockableTrait (lock/unlock with keyId), and SceneryTrait (can't be picked up). Room exits use the via property to check door state before letting the player through.

The player sequence is realistic: find key → unlock door with key → open door → walk through. The engine handles all validation — the story only declares the wiring.

About Sharpee

Sharpee is a TypeScript interactive fiction engine with a grammar-builder API, trait-based world model, and four-phase action pattern. This game demonstrates what's possible when the story layer fully engages the engine's extension points.
